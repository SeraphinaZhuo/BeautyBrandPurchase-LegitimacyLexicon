# METHODS.md — 方法说明书
**BPLL 项目 | 版本 v1.0 | 2026-07-08**

本文档汇总项目的全部方法规则。Research Brief 负责“为什么做、检验什么”；本文档负责“具体怎么做、按什么标准”。两份文档随仓库开源。

---

## 1. 分析原则

本项目在正式分析前固定以下方法原则。后续如需修改，只通过版本历史追加说明，不覆盖原文，以便追踪方法变更。

1. **分析单位是发言，不是用户**  
   本项目只分析公开文本中的“发言”及其话语特征，不推断具体用户的心理、身份或行为变化。因此，报告中不使用“某类消费者流失了”之类的个体层面因果表述；允许使用的表述是“某类发言的占比发生了变化”或“某类话语在特定时段中更常见”。

2. **结论只适用于公开发言人群**  
   语料来自关键词检索，因此只覆盖在平台上公开发声、且发言被关键词命中的人群。没有发声的人、未被关键词覆盖的讨论、私域讨论与线下态度，均不在本项目数据范围内。若要推断沉默人群或总体消费者态度，需要另行设计问卷或访谈研究。

3. **词典需通过验证后才进入统计分析**  
   词典自动标签不会直接用于正式结论。每一类词典都需要与人工标注结果对照，并公开报告 precision / recall 等指标。只有达到 §6 所列标准的类别，才进入自动化占比统计；未达标类别只用于质性阅读和案例引用。

4. **使用中性术语**  
   分析文本避免使用带有明显价值判断的修辞，例如“社交剥削”。本项目统一使用较中性的概念表述，包括“价值认同型”“社会信号型”“信任价值”“信号价值”等。

5. **普通用户与机构账号分开分析**  
   普通用户构成主样本。蓝V、媒体号、品牌官方账号等机构类账号单独形成机构样本，主要用于分析品牌回应、媒体传播或机构表态，不与普通用户发言混合计算。

6. **公开代码与聚合结果，不默认公开原文级数据**  
   本项目开源代码、词典、清洗规则与聚合统计结果。原文级数据是否公开，取决于平台服务条款、隐私风险与研究伦理要求；默认不公开可追溯到单条发言的原文数据。

7. **正式结果在方法固定后再运行**  
   涉及 T0 → T1 → T2 真实对比的统计分析，只在 Research Brief 冻结、词典完成验证之后执行。在此之前，管线开发仅使用测试数据或不会暴露真实前后变化的输入方式，以避免“先看结果、再调整方法”的问题。具体做法见 §7。


## 2. 数据清洗规则

数据清洗由 `clean_weibo.py` 执行。清洗过程不直接删除原始数据；被排除的记录会移动到归档文件，并保留 `exclusion_rule` 列，说明其被排除的原因。这样可以保证后续能够复查、恢复或重新运行清洗流程。

* **R1 去重**<br>
  如果 `post_id + 规范化文本` 完全重复，只保留首次出现的记录。<br>
  这类重复通常来自采集过程，而不是用户重复发言，因此不计入分析。<br>

* **R4 账号分流**<br>
  蓝V、媒体号、品牌官方账号等机构类账号会先被移入机构样本，再执行广告识别和低信息过滤。<br>
  这样处理是因为机构发帖、媒体报道和广告内容在表面特征上可能相似，例如都可能包含价格、活动、双 11 等信息。先进行账号分流，可以降低误删机构内容的风险。<br>

* **R2 广告识别**<br>
  广告识别采用规则打分，而不是简单关键词删除。<br>
  若文本命中强广告信号，例如“领券”“抽奖机制”“打款”“私我”等带货基础设施词，则直接判定为广告。<br>
  若只命中较弱信号，例如价格、大促词、行动号召等，则需要同时命中两类不同弱信号，或命中一个弱信号并出现模板化特征，才会被移出。模板化特征指同一 30 字前缀在 3 条及以上不同帖子中重复出现。<br>
  被识别为广告、但同时包含危机相关词的记录，会单独计数，便于后续人工复核。对应审计字段为 `AUDIT_promo_rows_containing_crisis_words`。

* **R3 低信息内容**<br>
  本项目不使用简单的长度阈值删除短文本，因为“不买”这样的短文本本身可能具有明确态度。<br>
  只有在文本有效长度不超过 1 个字，或全文命中已知噪音清单时，才会移出。噪音清单包括“转发微博”“关注”“图片评论”等明显缺乏可分析内容的文本。

* **R6 品牌词误命中**<br>
  对于“花王”样本，如果文本同时出现“牡丹”“芍药”“花中之王”“工笔”“书法”等语境，且没有日化产品相关语境，则判定为品牌词误命中并移出。<br>
  该规则主要用于排除“花王”作为普通词语或艺术语境出现的情况。

* **R7 分析标记**<br>
  部分记录不会被移出，但会额外添加标记列，供后续分析时分组使用。<br>
  `crisis_flag` 表示文本包含核污水相关词簇；<br>
  `huaxizi_flag` 表示蜂花样本中受到花西子事件影响的内容，在分析“国货红利”时需要单独列出；<br>
  `guohuo_flag` 表示文本包含国货相关表达。

* **R5 原始数据只读**<br>
  原始文件不在清洗过程中被修改。<br>
  任何人重新运行脚本，都应得到一致的输出结果。运行结果是否一致，以 `dq_stats.json` 的差异为准。<br>

## 3. 测量方法：话语类别占比

本项目不使用通用情绪分数作为核心指标。原因是，在本语料中，文本情绪和品牌态度经常不一致。例如，“干得漂亮”在字面上可能是正面表达，但在具体语境中可能是在支持抵制；“好用到纠结”字面上带有负面情绪，但可能表达的是对产品的认可。评论文本又普遍较短，中位数约为 13 字，因此通用情绪模型更容易误判。

因此，本项目采用“话语类别占比”作为主要指标。

* **核心指标**<br>
  在每一个“品牌 × 时段”的分析单元中，计算某一话语类别出现的发言数，占该单元全部发言数的比例。<br>
  例如，可以计算某品牌在 T1 时段中，“安全恐慌”类发言占全部发言的比例。<br>

* **类别体系**<br>
  话语类别分为两个层级。<br>
  第一层区分“价值认同型”和“社会信号型”话语。<br>
  第二层进一步区分七类危机反应：安全恐慌、立场抵制、理性辩护、摆烂虚无、割舍挣扎、替代转投和国货红利。<br>
  各类别的具体定义见编码手册。

* **允许多标签**<br>
  一条发言可以同时属于多个类别。<br>
  例如，一条评论可能既表达“安全恐慌”，也表达“替代转投”。因此，各类别占比是分别计算的，不要求加总为 100%。

* **置信区间**<br>
  每个占比都会报告 Wilson 95% 置信区间。<br>
  在最小分析单元中，样本量约为 129 条发言，对应的置信区间半宽约为 ±6–8 个百分点。因此，本项目更关注方向性差异和趋势变化，而不是过度解读小幅度的点估计差异。

* **稳健性检查**<br>
  除主分析外，本项目还提供两类补充结果。<br>
  第一类是点赞加权版本，用来观察高互动内容是否明显改变总体话语结构。<br>
  第二类是主贴与评论分开统计的版本，用来区分原始发帖和互动回应中的话语差异。<br>
  这些结果作为附录或补充输出提供。

* **与研究问题的对应关系**<br>
  M1 关注“立场抵制”占比在不同品牌之间的变化轨迹。<br>
  M2 关注“安全恐慌”占比在不同品牌之间的变化轨迹。<br>
  二者的相对强度用“立场抵制占比变化量 − 安全恐慌占比变化量”表示，即 Δ立场 − Δ恐慌。<br>
  机制 C 关注各品牌在 T0 阶段的第一层话语构成，是否与 T1 阶段的危机反应结构存在对应关系。由于品牌层面的样本量只有 5，该部分只做描述性展示，不做强推断。<br>
  机制 D 关注“理性辩护”类发言在互动结构中的位置。具体做法是查看这类发言挂靠在哪些 `parent_post_id` 下，以及它们引发了怎样的回应。

## 4. 种子词准入标准

本项目的词典不是直接由高频词自动生成，而是先根据人工精读和编码手册形成候选词，再用脚本检查这些词在语料中的实际出现情况。

具体顺序如下：

1. 精读部分语料，并在编码手册中记录类别判断；
2. 根据类别定义和精读结果，人工筛选候选种子词；
3. 使用 `audit_seed_words.py` 检查每个词在主语料中的出现频次；
4. 对低频或未出现的词进行公开标记，而不是悄悄删除。

这种顺序的目的，是避免把“出现频率高”误当成“类别判别力强”。也就是说，词典首先服务于类别定义，其次才通过语料频次进行检查。

一个词进入种子表，需要满足以下标准：

* **在语料中有根据**<br>
  优先收录在精读语料中真实出现过的表达。<br>
  如果某些词在后续频次检查中没有出现，或出现次数很少，不会直接删除，但会在词典中标记出来。

* **不是个体独特表达**<br>
  种子词应当在多个发言者、多个语境中出现。<br>
  只属于某一个用户的特殊说法，不进入种子词表。

* **优先保证判别力**<br>
  种子词阶段宁可少收，也不为了提高覆盖率而纳入含义模糊的词。<br>
  覆盖率不足的问题，后续通过词向量扩展和人工验证来处理。

* **优先使用自然表达**<br>
  词典尽量收录用户实际使用的说法，而不是研究者自己发明的分析术语。<br>
  例如，用户常说的“避雷”“不敢买”“换国货”，通常比抽象概念词更适合作为词典条目。

`audit_seed_words.py` 会自动检查每个种子词在清洗后微博主样本和小红书语料中的出现频次：

* 频次为 0 的词，标记为 `corpus_absent`；
* 频次为 1–4 的词，标记为 `sparse`；
* `corpus_absent` 和 `sparse` 合并视为 `prior_driven`。

`prior_driven` 词项会保留在最终词表中，并在词典文件里公开标记。<br>
这类词的作用主要是补足理论类别的覆盖范围，尤其是对危机语料中本来就较少出现的“社会信号型”表达。它们不被视为强语料证据。后续词典验证时，这些词是否真正适合对应类别，将由人工标注结果统一判断。

## 5. 消歧规则

有些词在不同语境下含义不同，不能只靠词本身判断类别。例如，“避雷”可能是在说产品不好用，也可能是在说因为核污水问题而避开日货。为减少误判，本项目对部分高风险词设置了上下文规则。

这些规则在 `tag_and_aggregate.py` 中执行。未列入下表的词，按照种子表中的 `layer` 和 `category` 直接归类。

| 词        | 默认归属         | 何时改判                              | 改判后归属   |
| -------- | ------------ | --------------------------------- | ------- |
| 避雷 / 踩雷  | L1 功效型（质量负评） | 上下文含“核”“辐射”“排海”“日货”“日系”“清单”等词     | L2 安全恐慌 |
| 烂脸       | L1 功效型       | 上下文同时出现“还买”“谁买”“活该”“应该”等表达，并且指向日货 | L2 立场抵制 |
| 囤货       | 不计入          | 上下文含“批次”“生产日期”“核”等词               | L2 安全恐慌 |
| 焦虑       | L2 安全恐慌      | 上下文含“别”“不要”“没必要”“制造”等否定焦虑的表达      | L2 理性辩护 |
| 检测       | L2 理性辩护      | 上下文含“检测仪”“自己测”或“买……测”等表达          | L2 安全恐慌 |
| 本子       | 不计入          | 上下文含“日本”“日货”“抵制”等词                | L2 立场抵制 |
| 随便 / 无所谓 | 不计入          | 上下文含“辐射”“核”“反正”“摆烂”等词             | L2 摆烂虚无 |

这张表会在词典 v1 定稿时固定。之后如果新增或修改规则，需要在版本历史中说明原因。

## 6. 词典验证标准

词典进入正式分析前，需要通过人工标注验证。验证的重点不是证明词典“完全正确”，而是判断每一类词典是否足够可靠，能否用于自动统计。

* **人工标注规模**<br>
  验证阶段抽取 500–800 条样本，由 3 名标注者独立标注。<br>
  标注一致性使用 Fleiss' κ 报告。<br>
  标注者是否纳入最终一致性计算，只依据其内部一致性表现，不根据其具体标注结果进行选择。

* **类别准入标准**<br>
  每一类词典单独评估 precision。只有当某一类别的 precision ≥ 0.75 时，该类别才进入自动化占比统计。<br>
  如果某一类别未达到这一标准，则降级为“人工辅助类别”。这意味着它仍可用于质性阅读、案例筛选和引用，但不输出该类别的占比统计。

* **覆盖率诊断**<br>
  本项目会报告主样本中至少命中一个 L2 标签的发言占比，用来观察词典覆盖了多少危机反应表达。<br>
  该指标不设置硬性门槛。但如果覆盖率低于 30%，报告中需要专门讨论可能的漏检来源，例如隐喻表达、反讽表达、缩写、平台黑话或新的流行说法。

* **precision 与 recall 的使用方式**<br>
  precision 用于判断某一类别能否进入自动统计。recall 用于判断词典是否漏掉了大量相关表达。<br>
  因此，precision 直接关系到类别是否“准许上线”；recall 更多用于解释结果边界和后续改进方向。


## 7. 正式运行规则

本节说明正式统计分析何时运行，以及如何记录运行过程。目的不是限制复现，而是避免在方法尚未固定时提前查看真实结果，从而影响后续规则设定。

- **默认不运行真实前后对比**  
  分析脚本默认不接受完整的 T0 → T1 → T2 真实数据输入。若要运行正式分析，必须显式加入 `--unseal` 参数。这样做是为了把开发测试和正式统计区分开。

- **开发阶段只使用测试输入**  
  在方法冻结前，管线开发使用两类测试输入：

  1. `--placebo shuffle`：随机打乱品牌标签，用于检查脚本是否能正常完成清洗、打标和汇总。随机种子固定为 42，便于复现。
  2. `--placebo t0-only`：只输入 T0 单一时段数据，用于检查单期统计结果是否正常生成，但不查看危机前后的变化。

- **正式运行会留下记录**  
  每次运行都会写入 `run_log.jsonl`，记录运行时间、运行模式、输入文件摘要哈希等信息。该日志用于说明哪些结果是在方法冻结前生成的测试结果，哪些是在方法冻结后生成的正式结果。

- **结果解释以正式运行为准**  
  测试输入产生的结果只用于检查代码和输出格式，不用于 substantive interpretation。正式报告中的统计结论，只基于通过 `--unseal` 执行、且写入运行日志的正式分析结果。

## 8. 人工标注流程

本节说明词典验证所需的人工标注如何组织，包括样本抽取方式、标注者安排和分歧处理方式。

* **标注样本的抽取方式**<br>
  * 标注样本从清洗后的微博主样本（6,646 条）中抽取，共 650 条，分为试标批次 50 条和正式批次 600 条。
  * 抽样不采用完全随机方式。原因是包含核污水相关词汇的发言（`crisis_flag = 1`）只占主样本的约 8%。如果完全随机抽取 600 条，预计只会抽到 50 条左右这类发言，不足以评估七个危机反应类别的标注质量。
  * 因此采用以下配额：每个批次的一半从 `crisis_flag = 1` 的发言中随机抽取，另一半从其余发言中按"品牌 × 时段"的实际比例分层抽取。试标批次先抽出，正式批次从剩余发言中抽取，两批不重叠。
  * 这一设计的代价是，标注样本有意提高了危机相关内容的比例，因此它适合用于评估词典的判断准确率（precision / recall），**不适合**用于估计各类话语在总体语料中的真实占比。占比估计只来自词典对全量主样本的打标结果。
  * 抽样由 `sample_for_annotation.py` 执行，随机种子固定，重复运行可得到完全相同的批次。

* **标注者安排**<br>
  * 正式批次由 3 名标注者独立标注。
  * 标注者只能看到发言文本和编号，看不到品牌归属、发布时段等元数据列。三名标注者拿到的条目相同，但排列顺序各自独立打乱，以避免顺序效应造成的一致性虚高。
  * 标注者不被告知本项目的研究假设，以避免标注向预期方向偏移。标注期间，标注者之间不讨论具体条目。

* **试标与定义修订**<br>
  * 正式标注前，三名标注者先共同标注同一批 50 条试标样本，随后召开一次讨论会，对照分歧、澄清标签定义。
  * 若编码手册或标注指南因此修订，修订记录写入版本历史。
  * 正式批次开始后，标签定义不再变动。试标批次的标注结果不进入最终人工标准。

* **最终标准的形成与分歧处理**<br>
  * 正式批次中，一条发言的某个标签以多数意见（三人中至少两人一致）为准。
  * 若三人意见完全分散、或多数意见为“无法判断”的条目，不进入最终标准，其数量在验证报告中如实报告。
  * 标注一致性使用 Fleiss' κ 逐类别报告。标注者是否纳入一致性计算，只依据其内部一致性表现判断，与其具体标注倾向无关。
